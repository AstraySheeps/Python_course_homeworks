# 模拟退火算法代码理解 (`src/algorithms/sa.py`)

---

## 一、整体结构

```
solve_sa(problem, seed)  ← 统一入口（与 GA/贪心接口一致）
    │
    └─► SimulatedAnnealing(problem).solve()
            │
            ├─► _initial_solution()     ← 用贪心算法生成初始解（路线列表）
            │
            ├─► problem.evaluate_solution()  ← 统一人民币成本评估
            │
            ├─► _generate_neighbor()    ← 随机选择一种邻域操作
            │       ├─► _neighbor_2opt()      ← 内部逆转子路径
            │       ├─► _neighbor_swap()      ← 内部交换两个客户
            │       ├─► _neighbor_relocate()  ← 跨机迁移一个客户
            │       ├─► _neighbor_exchange()  ← 跨机交换两个客户
            │       └─► _neighbor_oropt()     ← 跨机迁移连续2-3个客户
            │
            └─► Metropolis 接受准则 + 自适应冷却
```

---

## 二、超参数（config.py）

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `SA_INIT_TEMP` | 1000 | 初始温度，高温时接受劣解概率大，搜索范围广 |
| `SA_COOLING_RATE` | 0.98 | 降温速率 α，`T = T × 0.98`，越接近 1 降温越慢 |
| `SA_MIN_TEMP` | 0.01 | 终止温度，低于此值时停止搜索 |
| `SA_ITERATIONS_PER_TEMP` | 100 | 每个温度平台上的扰动尝试次数 |
| `SA_ADAPTIVE_PATIENCE` | 50 | 连续无改善步数阈值，触发加速冷却 |

温度下降过程：T 从 1000 开始，每轮 ×0.98，约需 `log(0.01/1000) / log(0.98)` ≈ 570 轮 × 100 次/轮 ≈ 57000 次评估后收敛。

## 三、核心类与方法

### 3.1 `SimulatedAnnealing.__init__()` — 初始化（第 21–28 行）

```python
class SimulatedAnnealing(BaseAlgorithm):
    def __init__(self, problem, config=None):
        super().__init__(problem, config)
        self.T0       = config.get('T0', SA_INIT_TEMP)        # 初始温度
        self.alpha    = config.get('alpha', SA_COOLING_RATE)  # 降温速率
        self.T_min    = config.get('T_min', SA_MIN_TEMP)      # 终止温度
        self.iter_per_T = config.get('iter_per_T', ...)       # 每温度迭代次数
        self.patience = config.get('patience', ...)           # 自适应耐心值
```

可通过 `config` 字典在运行时覆盖默认参数，便于参数敏感性实验。

---

### 3.2 `_initial_solution()` — 初始解生成（第 30–41 行）

```
输入：无（通过 self.problem 获取问题数据）
输出：routes ← list[list[int]]，路线列表
```

**逻辑**：
1. 按客户到仓库的距离升序排列 → `order`
2. 调用贪心解码器 `_decode_greedy(problem, order)`，将排列转为路线
3. 扩展到 m 架无人机（不足的补空列表）

> 使用贪心解作为初始解（而非随机解），保证 SA 从一个合法解开始搜索，加速收敛。

---

### 3.3 邻域操作（第 43–113 行）

SA 的搜索能力来自五种邻域操作，每次迭代随机选一种。操作直接修改**路线列表**（而非排列编码）。

#### 3.3.1 `_neighbor_2opt()` — 内部 2-opt（第 43–53 行）

```
输入：routes
输出：new_routes（某条路线的一段子路径被逆转）
```

随机选一架有 ≥2 个客户的无人机，在其路线内随机选一段 `[i, j]`，将这段顺序反转。

```
例：[A, B, C, D, E] → 选 i=1, j=3 → [A, D, C, B, E]
```

#### 3.3.2 `_neighbor_swap()` — 内部交换（第 55–65 行）

```
随机选一架有 ≥2 个客户的无人机，交换其路线内两个客户的位置。
```

#### 3.3.3 `_neighbor_relocate()` — 跨机迁移（第 67–83 行）

```
从一架非空无人机中随机取一个客户，插入到另一架无人机的随机位置。
```

#### 3.3.4 `_neighbor_exchange()` — 跨机交换（第 85–95 行）

```
选两架不同的非空无人机，各取一个客户，互相交换。
```

#### 3.3.5 `_neighbor_oropt()` — 跨机块迁移（第 97–113 行）

```
从一架有 ≥3 个客户的无人机中取连续 2-3 个客户（一个"块"），
整体移到另一架无人机的随机位置。
```

若没有满足条件的无人机，退化为 `_neighbor_relocate()`。

#### 3.3.6 `_generate_neighbor()` — 随机调度（第 115–125 行）

```python
def _generate_neighbor(self, routes):
    ops = [self._neighbor_2opt, self._neighbor_swap,
           self._neighbor_relocate, self._neighbor_exchange,
           self._neighbor_oropt]
    op = random.choice(ops)   # 五选一，等概率
    return op(routes)
```

### 3.4 `solve()` — 核心退火算法（第 127–176 行）

```
输入：无（全部数据来自 self.problem）
输出：(best_solution, best_cost, history)
  best_solution ← list[list[int]]，最优路线
  best_cost     ← float，最优人民币总成本
  history       ← list[(iteration, cost)]，收敛轨迹
```

**逐段逻辑**：

```
第 129 行     current = 贪心初始解
第 130-131    current_cost = 评估当前解的总成本
第 133-135    初始化 best_solution/best_cost = current
第 137 行     T = T0（初始温度 1000）
第 138 行     steps_no_improve = 0
第 139 行     history = [(0, best_cost)]

第 142 行     while T > T_min:                    ← 温度降到 0.01 以下停止
第 143 行         for _ in range(iter_per_T):      ← 每个温度平台迭代 100 次
第 144 行             neighbor = 随机邻域操作(current)
第 145-146            neigh_cost = 评估邻居成本
第 147 行             delta = neigh_cost - current_cost

第 149 行             if delta < 0 or random() < exp(-delta/T):
                          ← Metropolis 准则：
                            • delta < 0（邻居更优）→ 一定接受
                            • delta > 0（邻居更差）→ 以 exp(-Δ/T) 概率接受
                            （T 高时容易接受劣解，T 低时几乎只接受优解）

第 150-152              current = neighbor
第 153-156              if 比全局最优更好 → 更新 best

第 158 行                 steps_no_improve += 1    ← 本步没有改进最优解

第 160-161             total_iter++，记录历史

第 164-167         自适应冷却：
                      if steps_no_improve >= 50:    ← 连续 50 步没进展
                          T *= (alpha * 0.8)         ← 加速降温，跳出停滞
                          steps_no_improve = 0
                      else:
                          T *= alpha                  ← 正常降温
```

**Metropolis 准则**：`exp(-delta / T)` 的含义：

| delta | T = 1000 | T = 100 | T = 1 |
|-------|----------|---------|-------|
| +10（差 10 元） | 99.0% | 90.5% | 0.0045% |
| +50（差 50 元） | 95.1% | 60.7% | ≈ 0% |
| +200（差 200 元） | 81.9% | 13.5% | ≈ 0% |

高温时大胆跳跃逃离局部最优，低温时精细打磨逼近最优。

**自适应冷却机制**：若连续 50 步没有发现更好的解，说明可能陷入局部最优，此时将降温速率临时加速（α × 0.8），让温度更快下降以跳出当前区域。

### 3.5 `solve_sa()` — 模块入口（第 179–185 行）

```python
def solve_sa(problem, seed=None):
    if seed is not None:
        random.seed(seed)
        np.random.seed(seed)
    algo = SimulatedAnnealing(problem)
    return algo.solve()
```

接口与 `solve_ga`、`solve_greedy` 完全一致，注册在 `SOLVERS` 字典中，`main.py` 可统一调用。

## 四、解的表示

与旧版不同，v3.0 的 SA 直接操作**路线列表**而非排列编码：

```python
routes = [
    [2, 5, 3],    # 无人机0: 仓库→2→5→3→仓库
    [1, 4, 6],    # 无人机1: 仓库→1→4→6→仓库
    [7, 8],       # 无人机2: 仓库→7→8→仓库
]
```

邻域操作直接在路线层面修改，评估也直接调用 `problem.evaluate_solution(routes)`，不再需要解码步骤。

## 六、与其他算法的关系

| 层级 | SA | 贪心 | GA |
|------|-----|------|-----|
| 解的表示 | 路线列表 | 路线列表 | 排列编码 + 贪心解码 |
| 初始解 | 贪心算法结果 | 按距离排序构建 | 随机排列 |
| 搜索策略 | 单解迭代 + Metropolis 接受 | 确定性构造 | 种群进化（选择/交叉/变异） |
| 邻域操作 | 5 种路线级操作 | 无 | OX 交叉 + 洗牌变异 |
| 评估 | `problem.evaluate_solution()` | 同 | 同 |
| 收敛保证 | 降温到 T_min | 一次完成 | 200 代或早停 |

SA 比 GA 实现更轻量（不依赖 DEAP），调参更直观（温度 vs 种群/交叉/变异），适合快速迭代实验。
