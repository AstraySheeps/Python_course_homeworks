# 无人机配送路径规划 — 总体代码理解

面向程序员的项目代码快速理解文档。重点解释核心算法与数据流，可视化与 Web 界面从简。

---

## 一、项目概览

**问题**：N 架无人机从配送中心出发，为 M 个客户配送包裹。目标是最小化总飞行距离，受载重上限和单趟里程上限约束。可选时间窗约束（VRPTW）。

**三种算法**：贪心（构造基线）→ 遗传算法（元启发式）→ 模拟退火（元启发式），共享相同的编码/解码框架。

**代码量**：约 2000 行 Python（6 个源文件）。

## 四、统一编码方案（三种算法共用）

**核心思想**：将"客户访问顺序"编码为一个排列，解码器按顺序贪心打包成满足约束的路线。算法只负责搜索最优排列，约束处理交给解码器。

### 4.1 编码

```
个体 = [3, 7, 0, 12, 5, ...]  ← 客户索引 0..n-1 的一个排列
```

这是整个系统的核心抽象：**所有三种算法都在搜索最优排列**。

### 4.2 解码：`decode_individual()` （`drone_delivery_genetic.py` 第 69 行）

按排列顺序依次将客户装入当前趟次：

```
depart_time = 最早空闲无人机的就绪时刻
current_time = depart_time

for 每个客户 in 排列:
    new_load = 当前载重 + 该客户重量
    new_dist = 已走 + 去该客户 + 回配送中心

    if new_load ≤ MAX_CAPACITY and new_dist ≤ MAX_DISTANCE:
        if 启用时间窗:
            arrival = current_time + 飞行时间
            if arrival > due_time: penalty += PENALTY_WEIGHT × 超时量
            current_time = max(arrival, ready_time) + SERVICE_TIME
        装入当前趟次
    else:
        if 趟次为空: 强制单飞 + 惩罚  # 兜底，防死循环
        else: 结束当前趟次，开启新趟次
```

**惩罚公式**：`penalty = PENALTY_WEIGHT × (max(0, load - capacity) + max(0, dist - max_dist) + max(0, arrival - due_time))`

惩罚值 1e4 远大于正常距离（~500），进化中不可行解会被自然淘汰。

### 4.3 `FleetScheduler` 类（`drone_delivery_genetic.py` 第 39 行）

模拟多架无人机并发出动的时序调度器：

```python
self.available_at = [0.0] * num_drones  # 每架飞机"下次可用时刻"

assign(flight_distance, duration=None):
    drone = earliest(available_at)     # 选最早空闲的
    depart = available_at[drone]
    flight_time = duration or (flight_distance / speed)
    arrive = depart + flight_time
    available_at[drone] = arrive      # 更新就绪时刻
    return (drone, depart, arrive)

min_available():
    return min(available_at)          # 查询最早可用时刻（不修改状态）
```

## 五、三种算法

### 5.1 贪心算法 `drone_delivery.py`

**函数**：`greedy_assignment(clients, dist_matrix, max_capacity, max_distance, time_windows=None)`

**策略**：多趟贪心构造
1. 从配送中心出发，计算所有未分配客户到当前位置的**有效成本**
2. 有效成本 = 距离 + 时间窗感知权重（迟到权重 3.0，等待权重 0.1，换算为等效距离）
3. 按有效成本排序，逐个尝试装入，第一个满足载重+里程约束的入选
4. 装不下则试次近客户，直到全部试完才开启新路线
5. 兜底：若剩余客户单独都无法满足约束，强制逐人分配并警告

**特点**：O(n²) 快速构造，作为性能基线。

### 5.2 遗传算法 `drone_delivery_genetic.py`

**函数**：`run_genetic_algorithm(clients, dist_matrix, num_drones, time_windows=None)`

**流程**：

```
初始化：随机生成 200 个排列（=种群）
评估：每个排列 → decode_individual → 适应度 = 总距离 + 惩罚

进化循环（200 代）：
    选择：锦标赛（5 选 1）× 190 次 → 190 个子代
    交叉：相邻配对，80% 概率做 cxOrdered（有序交叉，PMX 风格）
    变异：每个个体 30% 概率做 mutShuffleIndexes（洗牌 2/n 比例索引）
    精英保留：上代最优 10 个直接进入下一代
    → 新一代 = 190 子代 + 10 精英

返回：名人堂最优个体解码为路线
```

**算子说明**：

| 算子 | DEAP 函数 | 效果 |
|------|----------|------|
| 选择 | `selTournament(tournsize=5)` | 随机取 5 个，留最优 |
| 交叉 | `cxOrdered` | 有序交叉，保证子代仍是合法排列 |
| 变异 | `mutShuffleIndexes(indpb=2/n)` | 以低概率洗牌部分位置 |

**超参数**（经网格搜索调优）：`POP_SIZE=200, NGEN=200, CXPB=0.8, MUTPB=0.3, TOURNSIZE=5, ELITE=10`

### 5.3 模拟退火 `drone_delivery_sa.py`

**函数**：`simulated_annealing(clients, dist_matrix, num_drones, t0, t_min, alpha, iter_per_t, time_windows=None)`

**流程**：

```
for 每次重启（默认 3 次）:
    随机生成初始排列 → 评估适应度
    T = 500（初始温度）

    while T > 0.01:
        for _ in range(50):  # 每个温度平台迭代 50 次
            邻居 = generate_neighbor(当前解)
            ΔE = 邻居适应度 - 当前适应度

            if ΔE < 0 or random() < exp(-ΔE / T):
                接受邻居  # Metropolis 准则

        T *= 0.98  # 指数降温

    保留本轮最优 → 与全局最优比较
返回：全局最优
```

**邻域算子**（`generate_neighbor`，三种随机选用）：

| 概率 | 算子 | 示例 |
|------|------|------|
| 40% | Swap 交换 | `[A,B,C,D] → [A,D,C,B]` |
| 30% | Reverse 反转 | `[A,B,C,D] → [A,D,C,B]`（反转子序列） |
| 30% | Insert 插入 | `[A,B,C,D] → [A,C,B,D]` |

**Metropolis 准则**：`exp(-ΔE / T)` — 高温时接受劣解概率大（探索），低温时几乎只接受优解（利用）。

## 七、时间窗约束（VRPTW 扩展）

**启用方式**：`--tw` 命令行标志，或 Streamlit 勾选"启用时间窗约束"

**数据生成**（`common.py:generate_time_windows`）：
```
earliest_arrival = dist_matrix[depot, client] / DRONE_SPEED
ready_time = earliest_arrival + random(0, 15)
due_time   = ready_time + random(15, 40)
```
确保每个客户的时间窗从配送中心出发是可达的。

**三种算法的时间窗处理**：

| 算法 | 处理方式 |
|------|---------|
| 贪心 | 候选排序中加入时间窗感知权重 `key = dist + (late×3.0 + wait×0.1) × speed` |
| 遗传 | `decode_individual` 追踪 `current_time`，早到等待，迟到累加惩罚 |
| 退火 | 复用遗传的 `decode_individual`，与遗传完全一致 |

## 八、关键设计决策

1. **排列编码而非直接编码路线**：避免处理"哪个客户属于哪个无人机"的复杂约束，解码器自动打包。
2. **软约束 + 大惩罚**：超重/超距/迟到不直接禁止，而是施加 1e4 惩罚，优化器自然规避，同时保证永远有可行输出。
3. **FleetScheduler 时序模型**：模拟多架无人机并发调度，时间窗模式下追踪每客户到达时刻。
4. **接口统一**：`run_greedy()` / `run_genetic()` / `run_sa()` 返回相同 dict 结构，`main.py` 中可互换。
