# 无人机配送路径规划项目完善记录

## 完善日期：2026-06-09 — 全面重构（依据最终实施方案）

### 重构背景

原项目在未完整建模的情况下直接开发，导致以下问题：
- 目标函数使用抽象权重（w₁, w₂）和大M惩罚系数，物理含义模糊
- 时间窗不区分客户类型，起始时间相同，缺乏调度意义
- 仅在单一场景（20客户）上单次运行，无统计显著性
- 缺少随机搜索基线和多种对比实验
- 代码结构平面化，模块间职责不清
- 未区分"提前到达等待"与"迟到惩罚"

本次重构依据 `docs/最终实施方案（单趟路径问题）.md` 进行全面修改。

### 二、统一人民币成本模型（核心改动）

**改动前**：目标函数使用抽象权重
```python
COST_WEIGHT_DISTANCE = 1.0   # w1 — 无物理单位
COST_WEIGHT_TIME = 10.0      # w2 — 无物理单位
PENALTY_WEIGHT = 1e4         # M — 无物理单位
cost = w1 * distance + w2 * makespan + M * violations
```

**改动后**：统一折算为人民币
```python
COST_PER_KM = 0.8                  # 元/km（飞行成本）
COST_PER_MIN = 1.0                 # 元/min（运营时间成本）
PENALTY_DELAY_PER_MIN = 20.0       # 元/min（延迟罚金）
PENALTY_OVERLOAD_PER_KG = 500.0    # 元/kg（超重罚金）
PENALTY_EXCESS_RANGE_PER_KM = 300.0  # 元/km（超航程罚金）

TotalCost = 0.8×TotalDistance + 1.0×TotalTime
          + 20.0×TotalDelay + 500.0×TotalOverload
          + 300.0×TotalExcessRange
```

**设计原则**：延迟罚金 20元/min 远高于运营成本 1元/min，算法天然优先满足时效要求。

**变动文件**：`config.py`（新增）、`src/models/problem.py`（新增）

### 三、时间窗建模升级

**改动前**：
- 所有客户统一随机时间窗范围
- 起始时间相同（基于仓库直达距离+随机偏移）
- 不区分客户类型

**改动后**：
- 三种客户类型：紧急(urgent, 20%)、普通(normal, 60%)、宽松(relaxed, 20%)
- 错峰起始时间：紧急08:00、普通09:00、宽松10:00
- 时间窗宽度与类型匹配：急30-45min、普60-120min、松120-240min
- 明确区分早到等待（计入运营时间成本）和迟到惩罚（20元/min）

**时间链计算**：
```
arrival = 飞到客户点的时间
wait = max(0, e - arrival)      # 早到等待
delay = max(0, arrival - l)     # 迟到延迟
start = max(arrival, e)         # 开始服务时间
depart = start + service_time   # 离开时间
```

**变动文件**：`config.py`、`data/generate_data.py`、`src/models/problem.py`

### 四、多场景数据生成

**改动前**：仅20客户单场景，无类型区分

**改动后**：
- 三种规模场景：小(10客户/3架)、标准(30客户/5架)、大(50客户/8架)
- 客户坐标：70%高斯分布(热区) + 30%均匀分布(边缘)
- 服务时间与需求量弱正相关：`service_time = base + 0.3×demand + noise`
- 单趟可行性自动校验：`sum(demand) ≤ 0.85 × M × C`
- 道路曲折系数 β=1.3 纳入距离计算
- CSV输出到 `data/output/`

**变动文件**：`data/generate_data.py`（新增）

### 五、算法实现重写

| 算法 | 改动前 | 改动后 |
|------|--------|--------|
| 贪心 | 单一策略 | 两种变体：最近客户优先 + 紧急优先 |
| 模拟退火 | 排列编码，3种邻域操作 | 路线编码，5种邻域操作（含or-opt），自适应冷却 |
| 遗传算法 | 排列编码，手动实现 | 排列编码（DEAP），OX交叉+逆转变异，精英保留+早停 |
| 随机搜索 | 无 | 新增：10000次随机排列+贪心解码 |

**关键变化**：
- SA从排列邻域改为路线邻域操作（内部2-opt/swap + 跨机relocate/exchange/or-opt）
- GA编码保持"先排列后分组"但解码器改为贪心分割（按载重约束自动分组）
- 新增统一基类 `BaseAlgorithm`
- 所有算法接口统一为 `solve() -> (routes, cost, history)`

**变动文件**：`src/algorithms/base.py`、`greedy.py`、`sa.py`、`ga.py`、`random_search.py`（全部新增）

### 数据生成：按场景自动调整需求范围

**问题**：标准场景（30客户/5架）下，总运力 100kg × 0.85 = 85kg，但 30 客户 × 平均 4.5kg ≈ 135kg，远超上限。原 fallback 逻辑仅将需求 ×0.6，仍不满足。

**修复**：`generate_scenario()` 先计算每客户最大平均需求 `limit / num_customers`，据此设定需求上限 `min(8.0, max_avg × 1.6)`，然后按比例映射到 [1, demand_hi]。标准场景需求上限自动降至 4.5kg，大规模降至 4.4kg。

**变动文件**：`data/generate_data.py`

---

### 可行性判断修正：时间窗改为纯软约束

**问题**：`Problem.evaluate_solution()` 的 `is_feasible` 检查 `total_delay_time < 1e-6`，导致 SA 仅 0.76 分钟延迟即被判不可行。但时间窗在建模时已设计为软约束（延迟通过罚金体现），不应作为硬可行条件。

**修复**：`is_feasible` 仅检查硬约束（载重 `total_overload < 1e-6`、航程 `total_excess_range < 1e-6`），时间窗延迟由罚金自然惩罚。

**变动文件**：`src/models/problem.py`

---

### 贪心解码器重写：Best-Fit + EDD 排序

**问题**：原贪心解码器（`_decode_greedy`）将客户按顺序装入路线，满载后换下一架，剩余客户全部塞入最后一架 → 严重超载 → 500 元/kg 罚金导致成本飙至 5 万+。

**修复**：改为两步：
1. **Best-Fit 装箱**：每个客户选择能装入且剩余容量最充足的无人机，确保负载均匀分布、零超载
2. **EDD 趟内排序**：每条路线按客户截止时间升序重排（Earliest Due Date），大幅减少延迟

**效果**：

| | 修复前 | 修复后 |
|------|------|------|
| 超载 | 严重（成本 5 万+） | 0 |
| 延迟 | 40.8 小时 | 0.49 小时 |
| 总成本 | 51,353 元 | 2,446 元 |

**变动文件**：`src/algorithms/greedy.py`
