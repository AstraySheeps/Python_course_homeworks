# 遗传算法代码理解 (`src/algorithms/ga.py`)

---

## 一、整体结构

```
solve_ga(problem, seed)  ← 统一入口（与 SA/贪心接口一致）
    │
    └─► GeneticAlgorithm(problem).solve()
            │
            ├─► _create_toolbox(problem)    ← 创建 DEAP 工具箱（注册算子）
            │
            ├─► _decode_permutation(problem, perm)  ← 核心解码：排列 → 路线
            │
            └─► 进化主循环（200 代）
                    ├─► select  ← 锦标赛选择（tournament size=3）
                    ├─► mate    ← 有序交叉 OX（概率 0.8）
                    ├─► mutate  ← 洗牌变异（概率 0.15, indpb=0.05）
                    └─► 精英保留 ← 保留最优 5 个个体直接进下一代
```

## 二、超参数（config.py）

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `GA_POP_SIZE` | 100 | 种群个体数量，每代 100 个排列 |
| `GA_N_GENERATIONS` | 200 | 最多进化代数 |
| `GA_CX_PROB` | 0.8 | 交叉概率，80% 的配对体会做 OX 交叉 |
| `GA_MUT_PROB` | 0.15 | 变异概率，每个个体有 15% 概率被洗牌变异 |
| `GA_TOURNAMENT_SIZE` | 3 | 锦标赛选择规模，每次从 3 个个体中选最优 |
| `GA_ELITE_SIZE` | 5 | 精英数量，每代保留最优 5 个直接进入下一代 |
| `GA_EARLY_STOP_GEN` | 50 | 早停代数，连续 50 代无改善则提前终止 |

### 2.1 每代 100 个个体，这个数字始终不变

种群大小 **始终是 100 个**，不会多也不会少。每一代的操作本质上就是"淘汰一批、繁殖一批、再补回 100 个"：

```
第 0 代（初始）：随机生成 100 个排列
    ↓
第 1 代：从上一代选 95 个做交叉变异，再从上一代复制 5 个精英 → 又是 100 个
    ↓
第 2 代：同样，95 后代 + 5 精英 = 100 个
    ↓
...一直循环，每代都是 100 个
```

**数字账**：
- `GA_ELITE_SIZE = 5` → 每代留 5 个精英席位
- `GA_POP_SIZE - GA_ELITE_SIZE = 100 - 5 = 95` → 锦标赛需要选出 **95 个**后代
- `95 后代 + 5 精英 = 100` → 种群数恒定

---

### 2.2 锦标赛到底选了多少人，怎么选的

锦标赛选择不是"开一场大型比赛"，而是**一个小擂台，反复比 95 次**。

```
每次擂台：
  从种群中随机拉 3 个人（GA_TOURNAMENT_SIZE=3）
  比谁的适应度（总成本）更低
  赢的那个人进入"后代名单"
  三个人都放回种群（同一个人可以被多次选中）

重复 95 次 → 凑齐 95 个后代
```

**类比**：就像从一个班级里反复抽签——每次抽 3 个学生，成绩最好的那个获得"繁殖资格"。抽完把三人放回去，再抽下一轮。总共抽 95 轮，得到 95 个获胜者。成绩好的个体可能被抽中很多次，成绩差的可能一次都选不上——这就是"优胜劣汰"。

---

### 2.3 "克隆"是什么意思

DEAP 中的 `toolbox.clone(ind)` 是**深拷贝**——复制出一个内容完全相同但存放在不同内存地址的新个体。

```
克隆前： offspring[i] 和 pop[0] 指向同一个列表对象
         改 offspring[i] → pop[0] 也变了 ❌

克隆后： offspring[i] 是 pop[0] 的独立副本
         改 offspring[i] → pop[0] 不受影响 ✅
```

**为什么不直接用赋值？** Python 中 `a = b` 只是让 `a` 和 `b` 指向同一个对象。交叉和变异会修改列表内容，如果不先克隆，就会"改子代时误伤父代"。更严重的是，精英个体可能被交叉到一起改掉，导致最优解丢失。

---

### 2.4 名人堂（Hall of Fame）怎么发挥作用

名人堂是一个**独立于种群之外的保险箱**，始终保存着从第 0 代到当前代的全局最优 5 个个体。

```
                          ┌─────────────────────┐
                          │   名人堂 (HallOfFame)  │
                          │   最多存 5 个全局最优  │
                          │   hof[0] = 第1名       │
                          │   hof[1] = 第2名       │
                          │   ...                 │
                          │   hof[4] = 第5名       │
                          └──────────┬──────────┘
                                     │
              每代结束时 update(pop) │ 每代开始前取精英
              把种群中的新优秀个体   │ 克隆 hof 中的个体
              送入名人堂             │ 补进下一代
                                     │
                          ┌──────────▼──────────┐
                          │      种群 (pop)       │
                          │   100 个个体          │
                          │   每代都在变化         │
                          └──────────────────────┘
```

名人堂的两个关键作用：

1. **精英保留**：每代结束时，把名人堂中最优的 5 个个体克隆一份，直接塞进下一代。这些精英不参与交叉变异，原封不动保留——确保"最好的基因不会因为一次倒霉的交叉就被毁了"。

2. **最终答案**：进化结束后，`hof[0]` 就是整个进化史上遇到的最优个体，直接用它作为最终解。

**举例**：假设第 10 代碰巧出了一个特别优秀的个体（成本 1000），第 11 代种群整体表现不佳（最优才 1200），但因为名人堂保存了第 10 代的那个优秀个体，它会作为精英进入第 12 代，不会被遗忘。

## 三、编码与解码

### 3.1 编码方案："先排列，后分组"

**染色体 = 所有客户索引的一个排列**（顺序编码）。

```
例：n=8 个客户，染色体 = [5, 1, 8, 3, 7, 2, 6, 4]
```

这种编码天然合法——任何 0..n-1 的排列都是一个有效个体，OX 交叉和洗牌变异都不会产生非法解。

### 3.2 `_decode_permutation()` — 解码器（第 19–46 行）

```
输入：
  problem  ← Problem 实例
  perm     ← list[int]，客户索引的排列

输出：
  routes   ← list[list[int]]，过滤掉空路线后的路线列表
```

**逐段逻辑**：

```
第 26 行     routes = [[] for _ in range(m)]   ← m 架无人机，初始全空
第 27 行     drone_idx = 0                      ← 从第一架无人机开始填

第 30 行     for ci in perm:                    ← 按排列顺序取客户
第 31 行         if drone_idx >= m:             ← 所有无人机都用完了
第 32 行             routes[-1].append(ci)       ← 强行塞进最后一架（超载由罚金处理）
第 33 行             continue

第 35-36         drone = problem.drones[drone_idx]
                 current_load = 当前无人机的已分配载重

第 38 行         if current_load + demand > capacity:  ← 装不下
第 39 行             drone_idx += 1                     ← 换下一架无人机
第 40-42             如果换完也超了 → 强行塞最后一架

第 44 行         routes[drone_idx].append(ci)   ← 装入当前无人机

第 46 行     return [r for r in routes if r]    ← 过滤空路线
```

**贪心分割策略**：按排列顺序依次将客户分配给当前无人机，当前无人机装不下下一个客户时就换下一架。这保证了：
- 每架无人机的载重 ≤ 20kg（除非所有无人机都满了，才强行塞入并产生超重罚金）
- 路线内部不检查时间窗和航程——这些由 `problem.evaluate_solution()` 统一评估并计入罚金

> 与贪心算法的 `_decode_greedy` 不同：GA 的解码器是简单的"顺序填充"，而贪心算法使用"最佳适应（Best-Fit）"策略——对每个客户选择剩余容量最充足的无人机。

## 四、DEAP 工具箱配置

### 4.1 `_create_toolbox()` — 算子注册（第 49–79 行）

```python
def _create_toolbox(problem):
    n = problem.n

    # 幂等注册（避免重复导入时报错）
    if "FitnessMin" not in creator.__dict__:
        creator.create("FitnessMin", base.Fitness, weights=(-1.0,))
    if "Individual" not in creator.__dict__:
        creator.create("Individual", list, fitness=creator.FitnessMin)

    toolbox = base.Toolbox()
```

DEAP 类型注册：

| 注册项 | DEAP 类 | 含义 |
|--------|---------|------|
| `FitnessMin` | `base.Fitness` | 适应度类型，`weights=(-1.0,)` 表示最小化问题 |
| `Individual` | `list` | 个体就是 Python 列表（客户排列） |

算子注册：

| 注册名 | 算子 | 说明 |
|--------|------|------|
| `individual` | 自定义函数 | `random.shuffle(range(n))` 生成随机排列 |
| `population` | `tools.initRepeat` | 重复生成个体构成种群 |
| `evaluate` | 自定义函数 | 解码 + 评估总成本，返回 `(cost,)` 元组 |
| `mate` | `tools.cxOrdered` | 有序交叉 OX，保证子代仍是合法排列 |
| `mutate` | `tools.mutShuffleIndexes` | 洗牌变异，`indpb=0.05` 表示每个位置有 5% 概率参与洗牌 |
| `select` | `tools.selTournament` | 锦标赛选择，`tournsize=3` |

**适应度函数**（第 69–72 行）：

```python
def evaluate(ind):
    routes = _decode_permutation(problem, ind)   # 排列 → 路线
    eval_result = problem.evaluate_solution(routes)
    return (eval_result['total_cost'],)          # 返回元组，DEAP 要求
```

## 五、进化主循环

### 5.1 一张图看懂每代发生了什么

把一代理解为"学生选拔考试"，每一轮要做 4 件事：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      第 N 代（100 个学生）                            │
│                                                                     │
│  ① 锦标赛选拔（选95人）                                              │
│     反复抽签：每次拉 3 人上擂台，总成本最低的晋级。重复 95 次。         │
│     好学生可能被抽中多次，差学生可能一次都选不上。                      │
│                              ↓                                      │
│  ② 繁殖变异（改造95人）                                              │
│     把这 95 人两两组队：                                              │
│       - 80% 的组做"OX交叉"→ 互相交换一部分基因                       │
│       - 15% 的人做"洗牌变异"→ 随机打乱自己的一部分基因               │
│     改造后这些人的"成绩单"作废（标记为失效，需要重考）。              │
│                              ↓                                      │
│  ③ 重考（评估失效者）                                                │
│     对所有被改造过的学生重新计算总成本。                               │
│                              ↓                                      │
│  ④ 补回精英（凑齐100人）                                             │
│     从名人堂取出上一代最优的 5 人（原封不动，没经历改造），             │
│     跟 95 个改造后的后代合并 → 又是 100 人。                          │
│     把新一届学生的成绩汇报给名人堂，名人堂更新它的 Top 5 排名。        │
│                                                                     │
│  → 第 N+1 代（100 个学生），回到 ①                                    │
└─────────────────────────────────────────────────────────────────────┘
```

**为什么精英要原封不动保留？** 交叉和变异可能把好的基因改坏。如果不保留精英，上一代好不容易找到的最优解可能在交叉中被打散、再也找不回来。精英保留就是"保底机制"——最优解永远不会丢失。

### 5.2 用具体数字走一遍完整流程

以 n=30 个客户、5 架无人机为例：

```
【第 0 代 — 初始】
  随机生成 100 个排列，比如:
    pop[0]  = [5, 12, 0, 3, 7, ..., 28]  → 解码评估 → 成本 2850 元
    pop[1]  = [18, 2, 9, 14, 1, ..., 6]  → 解码评估 → 成本 3100 元
    ...
    pop[99] = [22, 7, 15, 3, 0, ..., 11] → 解码评估 → 成本 4200 元

  全部评估完毕 → hof.update(pop)
  名人堂 Top 5: [成本2500, 2600, 2650, 2700, 2750]（按成本排序）
  history = [(0, 2500)]  ← 第 0 代最优成本

【第 1 代 — 进化】
  ① 锦标赛 95 次：
       第1次: 抽到 pop[42], pop[7], pop[81] → pop[7] 成本最低 → 晋级
       第2次: 抽到 pop[3], pop[56], pop[90] → pop[3] 成本最低 → 晋级
       ...重复 93 次...
       offspring = [pop[7]的克隆, pop[3]的克隆, ...]  共 95 个

  ② 交叉（80%概率 = 约38对会交叉）：
       pair1: offspring[0] 和 offspring[1] → 随机数 0.6 < 0.8 → 做OX交叉
       pair2: offspring[2] 和 offspring[3] → 随机数 0.9 > 0.8 → 跳过
       ...
       交叉过的个体，适应度标记为"失效"

  ③ 变异（15%概率 = 约14个人会变异）：
       offspring[5] → 随机数 0.12 < 0.15 → 触发洗牌变异
       offspring[5] = [5, 12, 0, 3, 7, ...] 
                     → 随机打乱部分位置 → [5, 0, 12, 7, 3, ...]
       适应度标记为"失效"

  ④ 重新评估所有"失效"个体

  ⑤ 精英补回：
       elites = [hof[0]的克隆, hof[1]的克隆, ..., hof[4]的克隆]
       pop = [改造后的95个后代] + [未改造的5个精英] = 100 个

  ⑥ hof.update(pop) → 名人堂可能刷新，比如第1代发现成本 2400 的新个体
  ⑦ history = [(0,2500), (1,2400)]
  ⑧ best_cost 更新为 2400, no_improve=0

【第 2 代, 第 3 代...重复】
  直到第 147 代时已经连续 50 代没有改进 → 早停，进化结束

【最终】
  hof[0] = [8, 22, 1, 15, 5, ...]  ← 整个进化史上最好的排列
  _decode_permutation(problem, hof[0])
  → routes = [[8,22,1,15,5,...], [3,10,18,...], ...]
  → evaluate_solution(routes) → total_cost = 1628.26 元
```

### 5.3 `GeneticAlgorithm.solve()` — 源码对应（第 92–162 行）

```
输入：无（全部数据来自 self.problem）
输出：(routes, best_cost, history)
  routes     ← list[list[int]]，最优路线
  best_cost  ← float，最优人民币总成本

**源码逐段注释**：

第 103-105   初始评估：评估全部 100 个个体，更新名人堂
第 108 行     history = [(0, 第0代最优成本)]
第 109-110    best_cost / no_improve = 0

第 112 行     for gen in range(1, 201):          ← 最多 200 代
                  ↓
第 114 行         选择：锦标赛选出 95 个个体（= 100 - 5 精英位）
                  每次从 3 个随机个体中选适应度最好的 → 克隆

第 118-122        交叉：相邻两两个体配对
                  每个配对有 80% 概率做 OX 有序交叉

第 136-137        精英保留：
                  elites = 上一代名人堂最优 5 个的克隆
                  pop = 95 个后代 + 5 个精英 = 100 个

第 139 行         更新名人堂 hof.update(pop)
第 140-141        记录本代统计 → history

第 144-148        早停检查：
                  if 本代最优 < best_cost:  → 更新 best_cost, no_improve=0
                  else:                     → no_improve += 1

第 151-152        if no_improve >= 50:  → break（提前终止）

第 155-162    解码最优解：
              best_ind = hof[0]                    ← 名人堂第 0 位 = 全局最优
              routes = _decode_permutation(problem, best_ind)
              评估并返回
```

def solve_ga(problem, seed=None):

## 六、`_decode_permutation` vs `_decode_greedy` — 两种解码器对比

| 维度 | `_decode_permutation`（GA/随机搜索用） | `_decode_greedy`（贪心/SA 初始解用） |
|------|----------------------------------------|--------------------------------------|
| 策略 | 顺序填充：按排列顺序逐个装，装满换下一架 | 最佳适应：每个客户选剩余容量最足的无人机 |
| 趟内排序 | 无 | EDD 排序（按截止时间升序），减少延迟 |
| 超载处理 | 换下一架，全满则硬塞最后一架 | 换下一架，全满则塞负载最轻的 |
| 位置 | `src/algorithms/ga.py` | `src/algorithms/greedy.py` |

## 八、为什么用排列编码？

这个问题天然适合排列编码：

- **简洁**：客户被依次访问的顺序就是排列顺序，不需要为"哪个客户属于哪个无人机"设计复杂编码
- **合法**：OX 有序交叉和洗牌变异都保证子代仍是 0..n-1 的有效排列，无需修复算子
- **解耦**：GA 只负责搜索最优访问顺序，解码器负责把顺序转化为具体路线
- **灵活**：通过改变解码策略（顺序填充 vs 最佳适应）即可得到不同算法变体

## 九、与其他算法的关系

| 层级 | GA | 贪心 | SA |
|------|-----|------|-----|
| 编码方式 | 排列编码 + 贪心分组解码 | 直接构造路线 | 直接操作路线列表 |
| 搜索策略 | 种群进化（100个体 × 200代） | 确定性构造 | 单解迭代 + Metropolis |
| 关键算子 | OX交叉 + 洗牌变异 + 锦标赛选择 | 最近优先/紧急优先 | 5种路线级邻域操作 |
| 全局搜索 | 强（种群多样性） | 弱（贪心无回溯） | 中（Metropolis + 自适应冷却） |
| 运行时间 | 较长（200代 × 100个体） | 极快 | 中等 |
| 第三方库 | DEAP | 无 | 无 |
