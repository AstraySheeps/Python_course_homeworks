  三、decode_individual() — 核心解码函数（第69-145行）

  这是整个GA最关键的函数。

  输入：

  ┌──────────────┬───────────────────────────────────────────────────┐
  │     参数     │                       含义                        │
  ├──────────────┼───────────────────────────────────────────────────┤
  │ individual   │ 一个客户索引的排列，如 [5, 12, 0, 3, ...]，长度=n │
  ├──────────────┼───────────────────────────────────────────────────┤
  │ clients      │ shape=(n,3) 的客户数据 [x, y, weight]             │
  ├──────────────┼───────────────────────────────────────────────────┤
  │ dist_matrix  │ (n+1)×(n+1) 的距离矩阵，第0行/列为配送中心        │
  ├──────────────┼───────────────────────────────────────────────────┤
  │ time_windows │ 可选 (ready_times, due_times)，None=不启用时间窗   │
  ├──────────────┼───────────────────────────────────────────────────┤
  │ service_time │ 每个客户点的卸货服务时间，默认 1.0 时间单位        │
  └──────────────┴───────────────────────────────────────────────────┘

  输出：

  ┌─────────┬──────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │  输出   │                                                 含义                                                 │
  ├─────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ routes  │ 路线列表，每条路线是一个 dict：drone_id, depart_time, arrive_time, route, load, distance, deliveries │
  ├─────────┼──────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ penalty │ 约束违反总惩罚值（超重 + 超距 + 迟到）                                                               │
  └─────────┴──────────────────────────────────────────────────────────────────────────────────────────────────────┘

  逐段逻辑：

  i = 0  ← 指向排列中"下一个待处理的客户"
  while i < n:                        # 外层循环：一趟新飞行
      route = [], load=0, dist=0       # 初始化当前趟次
      current_pos = 0                  # 从配送中心出发
      depart_time = scheduler.min_available()  # 最早可出发时刻
      current_time = depart_time       # 追踪累积时间（含等待+服务）

      while i < n:                     # 内层循环：逐个装客户
          client_idx = individual[i]   # 按排列顺序取客户
          计算：装进这个客户后，载重和总里程是否超限？

          若不超限：
            - 若有时间窗：
                arrival = current_time + 飞行时间
                若 arrival > due_time → 累加迟到惩罚
              current_time = max(arrival, ready_time) + service_time
            - 装入当前趟次，i++，继续
          若超限：
            - 当前趟次为空？→ 强制单飞（加惩罚），i++ 跳过
            - 当前趟次非空？→ break，结束本趟，开启下一趟

      把当前趟次记录到 routes（加入回程距离 + 以实际耗时调用 scheduler.assign）

  关键细节：

  - 第95-96行 dist_to_next 和 dist_back：client_idx + 1 是因为距离矩阵第0行是配送中心，客户索引需要 +1。
  - 第99-100行 预估距离公式：已走 + 去程 + 回程 —— 必须在装客户前就知道装完后能否飞回配送中心。
  - 第110-129行 兜底逻辑：如果一趟的第一个客户就超限，说明这个客户无论怎样都无法被正常服务（数据问题或参数不合理），强制分配并加惩罚，保
  证算法不会卡死。
  - 第123-127行 惩罚公式：10000 × (超重 + 超距) —— 权重很大，确保不可行解的适应度极差，进化中被淘汰。

  ---
  总结：为什么用排列编码？

  这个问题天然适合排列编码：
  - 客户被依次访问的顺序就是排列顺序
  - 解码器按顺序"贪婪打包"：前面的客户填满一趟，趟满了自动开下一趟
  - cxOrdered 交叉和 mutShuffleIndexes 变异都保证子代仍是合法排列
  - 避免了对"哪个客户属于哪个无人机/哪趟"的复杂离散约束处理

  核心思想是让遗传算法只负责搜索最优的客户访问顺序，解码器负责把顺序自动转化为满足约束的路线。
