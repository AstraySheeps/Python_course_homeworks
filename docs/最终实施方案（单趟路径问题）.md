## 二、问题建模（整合版）

### 2.1 场景描述

一个物流中心（仓库）拥有 M 架同型号无人机，每天需要向 N 个客户配送货物。**每架无人机只执行一趟配送任务**（单趟模式），因此数据生成时必须确保所有客户的总需求量不超过全部无人机的总载重。

每架无人机有：**最大载重 C**、**飞行速度 v**、**最大航程 D_max**。
每个客户有：**坐标 (x, y)**、**需求量 demand**、**客户类型 type**、**时间窗 [e, l]**、**服务时间 service_time**。

系统需要决定：**(a) 哪些客户由哪架无人机配送，(b) 无人机访问客户的顺序**，使配送总成本（人民币）最低。

> **单趟模式的前提条件**：∑ demand_i ≤ 0.85 × M × C（总需求不超过总运力的 85%），确保存在可行的单趟分配方案。数据生成脚本将校验此条件。

### 2.2 数学建模

#### 决策变量
x_{ijk} ∈ {0, 1} — 无人机 k 是否从客户 i 飞到客户 j。

#### 距离计算

```
dist_{ij} = β × √((x_i - x_j)² + (y_i - y_j)²)
```

其中 β = 1.3 为道路曲折系数（实际飞行路径与直线距离的比值）。

#### 时间窗处理逻辑（核心建模细节）

这是本方案与原始方案、改进建议最大的建模差异。VRPTW（带时间窗车辆路径问题）中，时间窗 [e, l] 限制的是**开始服务的时间**，而非到达时间：

```python
arrival_time   = 无人机飞到客户点的时间
start_service  = max(arrival_time, e)        # 不能早于时间窗开始
waiting_time   = max(0, e - arrival_time)    # 提前到达→等待
delay_time     = max(0, arrival_time - l)    # 晚到→延迟
```

| 情况 | 条件 | 处理 | 成本影响 |
|------|------|------|----------|
| 提前到达 | arrival < e | 等待至 e 才开始服务 | 等待时间计入运营时间成本（1 元/min） |
| 准时到达 | e ≤ arrival ≤ l | 立即开始服务 | 仅飞行时间计入运营成本 |
| 延迟到达 | arrival > l | 产生延迟 | 延迟时间按 20 元/min 罚金 |

> **为什么提前到达不算违规？**现实中，客户 9:00 上班，无人机 8:50 飞到，在门口等 10 分钟即可——提前到不罚款，但等待期间无人机和调度系统仍被占用，产生运营时间成本。

#### 服务时间与服务时序链

服务时间（service_time）是客户点上的**停留时间**，包含完整的现场作业流程：

```
无人机到达上空
    ↓
降落（约 0.5 min）
    ↓
卸货（取决于货物量）
    ↓
客户确认签收
    ↓
起飞（约 0.5 min）
    ↓
飞向下一个客户
```

在时间链中，服务时间紧随"开始服务"之后，直接影响后续客户的到达时间：

```
对路线 [c1, c2, c3, ...] 上的每个客户 ci：

  arrive_ci    = 从上一个点飞到 ci 的时间
  wait_ci      = max(0, e_ci - arrive_ci)          ← 早到等待
  start_ci     = max(arrive_ci, e_ci)
  depart_ci    = start_ci + service_time_ci         ← 服务完成后离开
  arrive_ci+1  = depart_ci + dist[ci][ci+1] / v     ← 飞往下一客户
```

> 服务时间通常与需求量正相关（货物越多，卸货越久），本方案取 `service_time = base + 0.3 × demand + noise`（5–15 min 范围）。这一细节虽小，但直接影响后续所有客户的时间窗判断——尤其是排在路线末尾的紧急客户，可能因为前面几站的累计服务时间而迟到。

#### 目标函数 — 统一人民币（元）

```
Minimize: TotalCost = 飞行成本 + 运营时间成本 + 延迟罚金 + 超重罚金 + 超航程罚金
```

| 成本项 | 系数 | 单位 | 说明 |
|--------|------|------|------|
| 飞行成本 | 0.8 | 元/km | 含电费、维护、折旧 |
| 运营时间成本 | 1.0 | 元/min | **飞行时间 + 等待时间 + 服务时间**，含调度管理 |
| 延迟罚金 | 20.0 | 元/min | 超出时间窗末尾的每分钟罚金（迟到） |
| 超重罚金 | 500.0 | 元/kg | 超出载重的每公斤罚金 |
| 超航程罚金 | 300.0 | 元/km | 超出最大航程的每公里罚金 |

> **系数设计原则**：延迟罚金 20 元/min 明显高于运营成本 1 元/min——少飞 25 km 仅省 20 元，晚到 1 分钟就罚 20 元。算法天然倾向于优先满足时效，符合现实物流企业"怕投诉甚于怕油费"的逻辑。

**完整公式：**

```
对于每架无人机 k 的路线 [c1, c2, ..., cm]：
  flight_time_k  = (dist[0][c1] + Σ dist[ci][ci+1] + dist[cm][0]) / v
  waiting_time_k = Σ max(0, e_ci - arrival_ci)
  delay_time_k   = Σ max(0, arrival_ci - l_ci)
  service_time_k = Σ service_time_ci
  
  operation_time_k = flight_time_k + waiting_time_k + service_time_k

TotalDistance   = Σ_k flight_time_k × v
TotalTime       = Σ_k operation_time_k
TotalDelay      = Σ_k delay_time_k
TotalOverload   = Σ_k max(0, Σ_{i in route_k} demand_i - C)
TotalExcessRange = Σ_k max(0, route_distance_k - D_max)

TotalCost = 0.8 × TotalDistance
          + 1.0 × TotalTime
          + 20.0 × TotalDelay
          + 500.0 × TotalOverload
          + 300.0 × TotalExcessRange
```

> 与原方案的对比：原方案用抽象权重 w₁、w₂ 和大 M 惩罚系数，单位不统一且未区分"等待"和"延迟"。本方案将所有项折算为人民币，并明确区分了提前到达（等，不计罚金但计时间成本）和延迟到达（罚），建模更精细、答辩时每个成本项都能解释其物理含义。

#### 约束条件

| 约束 | 公式 | 说明 |
|------|------|------|
| 容量约束 | Σᵢ (demandᵢ × Σⱼ x_{ijk}) ≤ C, ∀k | 每架无人机不超载 |
| 全局需求约束 | Σᵢ demandᵢ ≤ 0.85 × M × C | **数据生成的硬前提**，确保存在可行的单趟分配 |
| 时间窗约束（硬） | start_service_timeᵢ = max(arrivalᵢ, eᵢ) ≤ lᵢ | 开始服务时间不得晚于时间窗末尾 |
| 时间窗约束（软） | arrivalᵢ 无硬性下限 | 早到可以等待（等待时间计入运营成本），迟到产生罚金 |
| 航程约束 | Σᵢ Σⱼ (dist_{ij} × x_{ijk}) ≤ D_max, ∀k | 每架无人机不超最大航程 |
| 访问唯一性 | Σ_k Σⱼ x_{ijk} = 1, ∀i | 每个客户恰好被访问一次 |
| 流守恒 | Σⱼ x_{jik} = Σⱼ x_{ijk}, ∀i,k | 进入 = 离开 |
| 仓库约束 | Σⱼ x_{0jk} = 1, Σᵢ x_{i0k} = 1, ∀k | 从仓库出发，返回仓库 |

> **关于软约束 vs 硬约束**：容量、航程、访问唯一性、流守恒、仓库约束为**硬约束**（不可违反）。时间窗采用**软约束 + 惩罚**策略——迟到产生罚金，早到产生等待（运营时间成本）。元启发式算法（SA、GA）搜索中可能产生不可行解，目标函数中的高额罚金会自动引导搜索远离不可行区域。最终最优解应满足所有硬约束、且罚金为 0。

> **单趟可行性保证**：数据生成阶段即校验 `sum(demand) ≤ 0.85 × M × C`，确保在载重维度上存在可行的单趟分配方案。85% 而非 100% 的阈值留出余量，避免所有无人机都满载导致算法无优化空间。

### 3.2 仓库位置

```
仓库坐标: (25, 25)  （50×50 km 区域的中心）
```

### 3.3 客户坐标生成

- **70% 客户**：以仓库为中心的高斯分布（σ = 7.5 km），模拟"热区"客户
- **30% 客户**：区域内的均匀分布，模拟边缘/偏远客户

### 3.4 客户类型与时间窗

这是本方案相比原方案最重要的改进之一——**时间窗具有不同的起始时间**，使得提前到达→等待→迟到罚金的机制真正发挥作用：

| 客户类型 | 占比 | 时间窗所在时段 | 时间窗宽度 | 典型场景 | 示例 |
|----------|------|--------------|-----------|----------|------|
| 紧急订单 (urgent) | 20% | 08:00–09:00 | 30–45 min | 生鲜、药品、早餐外卖 | 08:00–08:45 |
| 普通订单 (normal) | 60% | 09:00–12:00 | 60–120 min | 常规快递、文件 | 09:30–11:30 |
| 宽松订单 (relaxed) | 20% | 10:00–16:00 | 120–240 min | 日用品、大宗包裹 | 10:00–14:00 |

**时间窗生成逻辑：**

```python
def generate_time_window(customer_type):
    if customer_type == "urgent":
        # 08:00-09:00 范围内，宽度 30-45 min
        tw_start = random.uniform(0, 0.75)      # 0~0.75 小时内（08:00-08:45）
        tw_width = random.uniform(0.5, 0.75)     # 30-45 min
    elif customer_type == "normal":
        # 09:00-12:00 范围内，宽度 60-120 min
        tw_start = random.uniform(1.0, 2.0)      # 1~2 小时内（09:00-10:00）
        tw_width = random.uniform(1.0, 2.0)      # 60-120 min
    elif customer_type == "relaxed":
        # 10:00-16:00 范围内，宽度 120-240 min
        tw_start = random.uniform(2.0, 4.0)      # 2~4 小时内（10:00-12:00）
        tw_width = random.uniform(2.0, 4.0)      # 120-240 min

    tw_end = min(8.0, tw_start + tw_width)        # 不超过工作日 8 小时
    return (round(tw_start, 2), round(tw_end, 2))
```

**为什么这样设计？**

```
时间线 →
08:00        09:00        10:00        12:00        16:00
 |-----紧急----|            |--普通-----------|            |----宽松-----|

紧急客户最早开放（08:00），无人机早到需等待至 08:00 才能开始服务
普通客户 09:00 开放，若无人机先送了紧急件再飞过来刚好赶上
宽松客户 10:00 开放，最晚可到 16:00，可灵活安排在路线末尾
```

这种错峰设计使得时间窗真正具有**调度意义**——算法需要权衡"先在仓库等到 08:00 送紧急件"还是"先飞远路送宽松件"。

## 四、算法设计

### 4.1 算法基类（统一接口）

```python
class BaseAlgorithm(ABC):
    def __init__(self, problem, config):
        self.problem = problem
        self.config = config
        self.best_solution = None
        self.best_cost = float('inf')
        self.history = []

    @abstractmethod
    def solve(self) -> tuple[list, float, list]:
        """返回 (最优解, 最优成本, 收敛历史)"""
        pass
```

### 4.2 贪心算法（基线）

**策略**：最近客户优先 + 约束检查。

```
1. 所有无人机从仓库出发
2. 对当前无人机：
   a. 从未访问客户中选择距离最近者
   b. 检查：载重 + 航程 + 时间窗（软约束，超时计入罚金但仍可加入）
   c. 通过 → 加入路径，更新位置
   d. 卡住（无客户能加入且不满足约束）→ 返回仓库，换下一架
3. 重复至所有客户被访问或无人机耗尽
4. 所有路径以返回仓库收尾
```

- 时间复杂度：O(n²)
- 角色：提供初始可行解，作为 SA 的初始解和性能下界。

**改进变体**：额外实现"优先紧急客户"版本（先排 urgent → normal → relaxed），对比两种贪心策略。

### 4.3 模拟退火算法

**初始解**：贪心算法结果。

**邻域操作**（每次迭代随机选择一个执行）：

| 操作 | 说明 |
|------|------|
| 内部 2-opt | 同一无人机内逆转一段子路径 |
| 内部 swap | 同一无人机内交换两个客户位置 |
| 跨机 relocate | 将一个客户从无人机 A 移到无人机 B 的某个位置 |
| 跨机 exchange | 交换两架无人机各一个客户 |
| 跨机 or-opt | 将一架无人机的连续 2–3 个客户整体移到另一架 |

**接受准则**：
- 新解更优 → 必接受
- 新解更差 → 以概率 exp(-Δ/T) 接受

**冷却策略**：
- 初始温度 T₀ = 1000，冷却率 α = 0.98
- 每个温度迭代 100 次
- 停止温度 T_min = 0.01
- 自适应：若连续 50 步无改善，α *= 0.8（加速冷却）
- 重启机制：记录全局最优；若降温至 T_min 时未改善，保留最优解

### 4.4 遗传算法

**编码方案**："先排列，后分组"。

- **染色体** = 所有客户的排列（顺序编码），例如 `[5, 1, 8, 3, 7, 2, 6, 4]`
- **解码** = 贪心分割：按排列顺序依次将客户分配给当前无人机，当加入下一个客户导致载重超限时，当前无人机返回仓库，换下一架。解码过程中同时检查航程约束。

**DEAP 配置**：

| 组件 | 方案 |
|------|------|
| 选择 | 锦标赛选择 (size=3) |
| 交叉 | 有序交叉 OX (概率 0.8) |
| 变异 | 逆转变异 + 交换变异 (概率各 0.05) |
| 精英保留 | 保留最优 5 个个体 |
| 种群大小 | 100 |
| 迭代代数 | 200 |
| 早停 | 若 50 代无改善则提前终止 |

**适应度函数**：直接使用目标函数 `TotalCost`（最小化，DEAP weights=(-1.0,)）。

### 4.5 新增：随机搜索基线

为验证三种算法的有效性，增加**随机搜索**作为第 4 种对比基线：生成 10000 个随机排列，用同样的解码器评估，取最优。预期其成本显著高于三种算法，验证"优化算法确实在起作用"。

| 消融实验 | GA: 去掉精英保留 / 去掉交叉 / 只用变异 | 验证各组件贡献 |

### 5.2 评价指标

| 指标 | 含义 | 报告方式 |
|------|------|----------|
| 总成本 (元) | 目标函数值 | mean ± std (30 次) |
| 总距离 (km) | 所有无人机的飞行距离之和 | mean ± std |
| Makespan (min) | 最后一个客户完成时间 | mean ± std |
| 算法运行时间 (s) | 单次求解耗时 | mean ± std |
| 可行率 | 30 次运行中产出可行解的比例 | 百分比 |
| 负载均衡度 | 各无人机载重的基尼系数 | 0–1，越小越均衡 |
| 最优性 Gap | (算法成本 - 已知最优) / 已知最优 | 百分比 |

### 5.3 统计检验

对主实验结果（30 次独立运行）进行：
- **配对 t 检验**（SA vs GA）验证差异是否显著（p < 0.05）
- **箱线图**展示各算法 30 次运行结果的分布
- **小提琴图**展示成本分布形态

## 十三、风险预案

| 风险 | 概率 | 应对措施 |
|------|------|----------|
| 总需求超过总运力 85% 导致无可行单趟解 | 中 | 数据生成脚本内置校验 `sum(demand) ≤ 0.85 × M × C`，若不满足自动重新生成；极端情况下可调整需求范围为 [1, 5] kg |

| 维度 | 原方案 | 最终方案 | 改进理由 |
|------|--------|----------|----------|
| 成本模型 | 抽象权重 w₁, w₂ + 惩罚系数 M | 统一人民币成本（元） | 物理含义清晰，答辩易解释 |
