## 三、逐章节写作指引

### 摘要（~200字）

模板：
> 本文研究无人机集群配送路径规划问题，针对多无人机、多客户点、多时间窗场景，以最小化统一人民币总成本为优化目标，在载重、航程和时间窗约束下设计并实现了四种求解算法。贪心算法作为快速基线；模拟退火通过5种邻域操作和自适应冷却搜索最优解；遗传算法采用排列编码和精英保留策略进行种群进化；随机搜索作为对比基线。实验结果表明，模拟退火取得最优成本（1,443元），遗传算法次之（1,628元），相较贪心基线（2,446元）分别降低41%和33%，随机搜索成本为贪心的8.6倍，验证了优化算法的有效性。本文的算法框架和对比分析方法可为低空物流路径优化提供参考。

### 4. 算法设计（~1800字，核心章节）

#### 4.1 编码方案与解码过程（~400字，三种算法共用）

**这是报告的亮点——统一编码框架**：

> 三种算法共享相同的解表示方式：将N个客户索引(0,1,...,N-1)编码为一个排列π，解码时按排列顺序贪心地将客户逐次装入各趟次，满足约束才装入，否则开启新趟次。这种编码方式天然保证每个客户恰好被访问一次，将约束检查集中在解码阶段，使算法设计者可专注于排列空间的搜索。

- 放一个编码→解码的流程图
- 放数学描述：解码算法伪代码

#### 4.2 贪心算法基线（~300字）

- 选择策略：从当前位置选最近且满足约束的未分配客户
- 趟次管理：当前趟无客户可装时结束本趟，开启新趟
- 时间复杂度：O(n²)
- 作为性能对比基线

#### 4.3 遗传算法（~800字）

**编码**：排列编码（已在4.1节说明）

**适应度函数**（直接使用统一人民币成本）：
```
TotalCost = 0.8×距离 + 1.0×运营时间 + 20.0×延迟 + 500×超重 + 300×超航程
```
所有项统一为人民币（元），物理含义清晰可解释。

**选择算子**：锦标赛选择（tournament size=5）
- 每组随机选5个个体，最优者胜出进入下一代

**交叉算子**：有序交叉（OX, Ordered Crossover）
- 保留父代排列中客户的相对顺序

**变异算子**：索引洗牌（Shuffle Indexes）
- 以概率 p_m=2/n 随机交换排列中的相邻元素

**精英保留**：每代表现最优的10个个体直接复制到下一代

**超参数**：
| 参数 | 值 | 选择依据 |
|------|-----|---------|
| 种群规模 | 200 | 网格搜索{100,200,300} |
| 迭代代数 | 200 | 收敛曲线在200代前已平坦 |
| 交叉概率 | 0.8 | 网格搜索{0.7,0.8,0.9} |
| 变异概率 | 0.3 | 网格搜索{0.1,0.2,0.3} |
| 锦标赛规模 | 5 | 网格搜索{3,5,7} |
| 精英数量 | 10 | 网格搜索{5,10,20} |

#### 4.4 模拟退火算法（~700字）

**基本思想**：模拟金属退火过程，高温时大范围搜索（接受差解），降温后局部精调。

**邻域结构**（三种算子随机选择）：
1. **Swap**（40%概率）：随机交换排列中两个位置
2. **Reverse**（30%概率）：反转一段子序列（2-opt风格）
3. **Insert**（30%概率）：将一个元素移到新位置

**Metropolis接受准则**：
```
P(接受) = 1            , 若 ΔE ≤ 0
        = exp(-ΔE/T)  , 若 ΔE > 0
```
其中ΔE=新解成本-当前成本，T为当前温度。

**降温策略**：指数降温 T ← α × T，α=0.98

**多次重启**：执行3次独立搜索，取全局最优，避免单次搜索陷入局部最优。

**超参数**：
| 参数 | 值 |
|------|-----|
| 初始温度 T₀ | 500 |
| 终止温度 T_min | 0.01 |
| 降温速率 α | 0.98 |
| 每温度迭代 | 50 |
| 重启次数 | 3 |

**伪代码**：放模拟退火主循环的简化伪代码

#### 5.3 遗传算法进化分析
- 进化曲线分析：种群最佳适应度随代数快速下降，约在100代后收敛
- 平均适应度波动原因：变异带来的种群多样性

#### 5.4 模拟退火搜索轨迹分析
- 3次重启均能找到接近最优的解（说明算法稳定性好）
- 高温段曲线波动大（广搜），低温段趋于平稳（精调）
- 分析温度参数α对收敛速度的影响

#### 5.5 算法适用性讨论
- 贪心：速度最快，适合实时性要求高的场景，但解质量一般
- 遗传：适合大规模问题，可并行化，但需要精心调参
- 模拟退火：实现简单，效果稳定，适合中等规模问题
